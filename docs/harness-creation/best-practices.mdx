# Best Practices

## Keep skills atomic

Each skill should do one thing. Resist the temptation to build a large skill that handles many related tasks — prefer composing several small skills at the harness level instead.

**Instead of:**
```
skills/
└── content/         # drafts posts, generates images, scans social, emails stakeholders
```

**Prefer:**
```
skills/
├── create-blog-post/
├── generate-images/
├── scan-social/
└── email-stakeholders/
```

Atomic skills are easier to test, easier to reuse across harnesses, and easier to update independently.

---

## Put cross-skill knowledge in references, not in skills

If a piece of information — a style guide, an API overview, a brand voice document — applies to more than one skill, it belongs in `references/`, not inside any individual skill.

Duplicating it across skills creates maintenance problems: when the style guide changes, you have to update it in multiple places.

---

## Keep HARNESS.md as short as possible

`HARNESS.md` is loaded into the agent's context window on every activation. Every line costs tokens. Write the minimum body that lets the agent understand the harness structure and navigate to the right skill or reference.

A good body:
- One short paragraph describing the agent's role
- A bulleted list of skills with one-line descriptions
- A bulleted list of references with one-line descriptions

A bad body:
- Repeats information already in skill or reference files
- Explains how skills work in general (the agent knows this)
- Contains lengthy prose that could live in a reference file

---

## Use subdirectories for large harnesses

When a harness grows beyond ~8–10 skills or references, organize them into named subdirectories. Reflect that structure in `HARNESS.md` so the agent doesn't have to traverse the filesystem to understand what's available.

```
skills/
├── data-access/
│   ├── query-database/
│   └── read-spreadsheet/
└── reporting/
    ├── generate-chart/
    └── write-summary/
```

---

## One harness per agent role

Don't add unrelated capabilities to a single harness because it's convenient. If your system needs a data analyst and a copywriter, build two harnesses and two agents. This keeps each agent's role bounded and its `HARNESS.md` legible.

---

## Share skills and references, don't copy them

When multiple agents need the same skill (e.g., `query-database`), maintain one copy and reference it from multiple harnesses. Copying creates drift.

A common pattern is a shared library directory alongside your harnesses:

```
project/
├── shared-skills/
│   └── query-database/
├── analyst-harness/
│   ├── HARNESS.md
│   └── skills/ -> symlink or copy of query-database
└── reporter-harness/
    ├── HARNESS.md
    └── skills/ -> symlink or copy of query-database
```

---

## Write descriptions first

Before writing the body of any `HARNESS.md`, `SKILL.md`, or reference file, write the `description` field. If you can't describe the thing in one sentence, the scope is too broad.

---

## Version your harness with git

Harnesses are plain files. Put them in version control. Tag releases. This lets you roll back when a change to `HARNESS.md` or a skill breaks agent behavior.
